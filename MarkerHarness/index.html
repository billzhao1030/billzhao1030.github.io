🎯 How it works — four steps from PDFs to signed grades
01

Drop your inputs

Student files in assignments/<name>/inputs/submissions/. Filenames need only {name}_anything.pdf; everything after the first underscore is ignored, so an LMS export drops in unchanged.

# scaffold + drop files
$ python -m markerHarness.pipeline init -a example
$ cp ~/Downloads/submissions/* markerHarness/assignments/example/inputs/submissions/
02

Configure (automatic)

The config_builder agent parses Markdown rubric tables into structured rubric.yaml. A SHA256 hash of your Markdown keeps it from re-running needlessly.

# auto-runs as part of `all`
$ python -m markerHarness.pipeline configure -a example
[configure] running config_builder (vllm:Qwen3-VL-4B-Instruct) ...
[configure] ✓ rubric.yaml generated → assignments/example/config/rubric.yaml
03

Grade with two agents

Scorer + Reviewer agree, revise, or flag for human review.

$ bash scripts/serve.sh start
$ python -m markerHarness.pipeline grade -a example
[grade] (3/3) smithjohn  scorer 6.4s · reviewer 1.8s · verdict=reasonable · total=87.3
[grade]   └─ |Δ|=0.3 ≤ tol=5 → converged
04

Moderate and report

Curve scores to the target overall mean, write a CSV with per-criterion columns, a Markdown report, and per-student feedback.

$ python -m markerHarness.pipeline moderate -a example
N=3  curved: 3
target mean=80 σ=8 → method=linear
achieved mean=80.0 σ=8.97
$ python -m markerHarness.pipeline report -a example
[report] ✓ wrote final_grades.csv / report.md / feedback/
🔌 Multi-backend — any MLLM you have access to

Each agent picks its backend in agents.yaml. The Scorer and Reviewer must share the same loaded model on the local GPU (single-instance constraint); other agents are free to use cloud APIs.

agents:
  scorer:
    backend: vllm            # local Qwen3-VL-4B for throughput
    temperature: 0.0
  reviewer:
    backend: vllm
    temperature: 0.7
    cot_prompt: true
  evolver:
    backend: anthropic       # stronger judge — dodges same-model bias
    model: claude-sonnet-4-6
    api_key_env: ANTHROPIC_API_KEY
  config_builder:
    backend: openai
    model: gpt-4o-mini       # cheap + good at parsing tables

Where the keys come from

  • OPENAI_API_KEY
  • ANTHROPIC_API_KEY
  • GEMINI_API_KEY (or GOOGLE_API_KEY)
  • Keys live in environment variables only — never in YAML, never in the repo.

Structured-output mechanisms (handled per-backend)

  • vLLM → extra_body.guided_json with brace-balanced fallback parsing
  • OpenAI → response_format=json_schema
  • Anthropic → tool-use (a synthetic tool whose input_schema = the JSON schema)
  • Gemini → response_mime_type=application/json + response_schema (needs pip install google-generativeai)
🧬 Filename convention + name matcher algorithm

Only the part before the first underscore is the student name token. Everything after (LMS-style ids, original filename) is ignored entirely.

smithjohn_12345_67890_FinalReport.pdf
└──name───┘└──────── ignored ────────┘

Same-student spelling variants — different word order, different case, extra whitespace — resolve to the same submission token. Useful whenever a typed name has to match a filename. Behind the scenes:

  1. Normalize both sides: NFKD-decompose, strip combining marks, casefold, keep only [a-z0-9] + CJK ideographs, concatenate.
  2. Exact match on normalized form.
  3. Permutation match: try every token ordering of the user input (up to 4 tokens) joined and compared.
  4. Containment: all user tokens must appear as substrings (suppressed when input < 4 chars).
  5. Levenshtein (fuzzy): distance ≤ 2 with length difference ≤ 2 (only when steps 1–3 found nothing).
  6. Ambiguity is never silent: multi-candidate matches return Ambiguous and surface in the reconciliation table.

CJK names like 张三 are kept verbatim — we don't auto-romanize. Try the demo above to see this in action.

📂 Outputs — every file the pipeline writes

Per assignment, under assignments/<name>/:

work/
  manifest.json                # ingest output: name tokens, resolved PDFs, flags
  extracted/                   # zip/rar/tar.gz contents
  converted/                   # docx→pdf outputs (if LibreOffice present)

results/
  raw_scores.json              # original scorer + reviewer + debate (immutable)
  moderation.json              # curve params + before/after + warnings
  final_grades.csv             # 1 row per student, columns:
                               #   name, final, band, special, status,
                               #   total_raw, total_reviewed,
                               #   <criterion-id>_score, <criterion-id>_comment   (per rubric)
                               #   flags
  report.md                    # flags top + distribution + warnings + grade table
  feedback/<name>.md           # per-student criterion-by-criterion writeup
  run_<ts>/<name>.json         # per-student blackboard: full debate audit, evolver input

The triple raw → reviewer-suggested → final is kept in three separate columns of the CSV so any moderation step is auditable.

🧪 Self-evolver — how the harness improves itself

Across runs, the Reviewer accumulates disagreements with the Scorer. The evolver agent reads those signals and proposes a new Scorer prompt — surgical edits only.

Signals consumed

  • median |Δ| between scorer total and reviewer suggested
  • per-criterion mention frequency in reviewer notes
  • top recurring reviewer complaints
  • systematic bias direction (scorer-high / scorer-low / mixed)

Safety gates (any failure → refuse to apply)

  • Surgical-edit floor: rejects proposals that rewrite > 40% of the prompt.
  • Same-model gate: refuses to run unless --allow-same-model is set (since evolver-as-scorer-judge shares blind spots).
  • Dry-run by default: prints the diff and rationale but doesn't activate.
  • Provenance: every proposed change must cite a specific signal in the rationale file.

Versioning + rollback

# dry-run — see what it would propose
$ python -m markerHarness.pipeline evolve -a example --dry-run --allow-same-model

# commit (creates prompts/scorer_v(N+1).md + history/rationale_v(N+1).md)
$ python -m markerHarness.pipeline evolve -a example --apply

# revert
$ python -m markerHarness.pipeline evolve -a example --rollback

The active prompt is tracked in prompts/active.json with a history list — every change is reversible.

📦 System dependencies (optional but useful)

The harness keeps going even when these are missing — affected submissions get flagged.

  • LibreOffice (soffice) — converts .docx / .doc submissions to PDF. Missing → those submissions flagged, pipeline continues.
  • rarfile + unrar/7z — for student .rar archives. pip install rarfile + system package.
  • google-generativeai — required only if you want the Gemini backend. pip install google-generativeai.
  • vLLM — needed only if you want a local backend. Cloud-only setups don't need a GPU at all.
⚠ Honest caveats — read before trusting the numbers
  • The default local scorer is a 4B model. Its absolute scores need calibration against human-graded samples — that's why grades are proposed, not signed.
  • Same-model self-review (scorer reviewed by the scorer's own backend) shares blind spots. Configure the evolver — or even the reviewer — to a stronger backend (Claude / GPT) to dodge that.
  • Forcing a target mean while locking some students to HD has real consequences for the rest of the class; the moderator surfaces the warning rather than hiding it.
  • Long PDFs (image-heavy or scanned) can blow the 32K context. Default content_mode is text-primary; tune max_pages for your assignment.
  • The harness produces proposals, not final grades. Teacher sign-off is always required.
🗺️ Roadmap — what's coming

The current release (v0.3) grades PDF / text reports end-to-end with multi-agent argumentation and a self-evolving scorer prompt. Future releases extend the surface significantly.

v0.4 · Multi-modal inputs

  • Native PowerPoint ingestion (slide-by-slide content + layout via python-pptx)
  • Video presentations: keyframe sampling + audio transcription, graded as a single multi-modal exhibit
  • Source-code submissions: language detection + syntax-aware chunking + optional execution

v0.5 · Tool use + tool registry

  • Agents can call tools mid-grading: web search for academic claims, code execution for student code, library / dataset lookup
  • Tools become a registerable concept: name, description, JSON schema, executor
  • Tool I/O joins the audit trail alongside the scorer↔reviewer debate

v0.6 · Validation loop

  • Pre-commit validators run before grades are sealed
  • User-customizable checks: rubric arithmetic consistency, distribution sanity, custom invariants, no-silent-F rules
  • A failing validator blocks apply — never a silent ship

v0.7 · Teacher-feedback evolution

  • Today the evolver consumes only the Reviewer's signal — same model, same blind spots
  • Tomorrow: the teacher's manual overrides on flagged students feed back as ground-truth signal
  • Closes the loop: human-in-the-loop learning, not just same-model self-review

v1.0 · Rebrand

"Harness" was honest for v0.x — a thin orchestration around a single model. With multi-modal inputs, tool use, validation, and teacher-feedback learning it stops being a harness. A new name is on the table; the audit-first, multi-agent, model-agnostic ethos stays.

quickstart

Six lines from clone to signed grades.

# one-time setup
$ conda create -n mllm python=3.12 -y && conda activate mllm
$ pip install -r requirements.txt

# scaffold an assignment
$ python -m markerHarness.pipeline init -a example
# drop description.md + rubric.md + submissions/ into assignments/example/inputs/

# run — `all` chains configure → ingest → grade → moderate → report
$ bash scripts/serve.sh start
$ python -m markerHarness.pipeline all -a example
$ bash scripts/serve.sh stop   # release the GPU

# outputs in assignments/example/results/
$ ls assignments/example/results/
final_grades.csv  moderation.json  raw_scores.json  report.md  feedback/  run_<ts>/

Don't have a GPU? Set backend: anthropic (or openai) in agents.yaml for every role — the pipeline works without vLLM. Recommended for the self-evolver in any case, to avoid same-model bias.
